# Питч для банка (RU) — управляемое ИИ-андеррайтинг-решение

> Каркас повествования для презентации про ИИ-консалтинг банкам. Демо («Кредитный конвейер») — это наглядное доказательство **Шага 3** метода FDE: «выкатить пилот за детерминированным шлюзом». Демо-данные **синтетические, обрабатываются локально, ничего не отправляется**. Регуляторные тезисы ниже опираются на [uzbekistan-regulatory-briefing.md](uzbekistan-regulatory-briefing.md) — перед обязательствами перепроверить с банком/юристами (см. «Открытые вопросы» там).

## 1. Позиционирование (10 секунд)
Я — Forward Deployed Engineer. В корпоративном ИИ сложная часть не модель, а **внедрение**: данные, соответствие регуляторам, доверие стейкхолдеров, продакшн. Мой принцип: **модель предлагает — решение принимает человек — каждый шаг в журнале аудита**. Цикл «наблюдать → оценивать → исправлять» и есть продукт, а не сам агент.

## 2. Проблема банка (не стратегия, а внедрение)
Инфраструктура есть, данные есть — но ИИ не в продакшене. Трудоёмкие процессы (андеррайтинг, комплаенс, AML) всё ещё ручные. Нет слоя оценок (evals), нет аудита моделей, нет управляемых шлюзов. Это ровно то, что я закрываю — на одном узком, ценном процессе.

## 3. Что показывает демо
Один кредитный процесс «от заявки до журнала аудита», два сюжета:
- **МСБ (оборотный кредит):** DSTI 34% — «в пределах политики» → человек передаёт на решение → проект оферты.
- **Физлицо (потребительский):** DSTI 87% — «вне политики» → человек отклоняет → проект уведомления об отказе с основанием.

Каждый этап: заявка → **контроль KYC/AML** → комплектность → **индикатор доступности (DSTI)** → **шлюз решения (человек)** → результат → **журнал аудита** с полной трассировкой. Индикатор — детерминированный и объяснимый (причины + ссылки на документы-доказательства), **не «чёрный ящик»**.

## 4. Три регуляторных соответствия (ядро питча)
1. **ИИ-закон → человек в контуре обязателен.** Поправки к закону «Об информатизации» (одобрены Сенатом 01.11.2025) запрещают принимать решения, затрагивающие права человека, **исключительно ИИ** — нужно участие человека. Кредитное решение — ровно такой случай. Наш **шлюз «решение принимает человек» — это соответствие по построению.** *(Дату вступления в силу уточнить.)*
2. **DSTI → индикатор привязан к реальному порогу ЦБ.** Порог долговой нагрузки ЦБ РУз — **50%** (с 01.01.2025), считается по данным кредитного бюро, срабатывает от 10 млн сум. В демо «вне политики» = DSTI > 50% — тот же порог. Мы не изобретаем скоринг, а делаем **прозрачным и аудируемым** применение уже действующей нормы.
3. **Данные → локальное развёртывание = наименьшее трение.** После смягчения локализации (27.03.2026) трансграничная обработка требует адекватности/оговорок + инфобезопасности + надзора. **Развёртывание внутри страны снимает эту нагрузку целиком.** Плюс **биометрия (MyID), генетические и телеком-данные обязаны оставаться в РУз** — а удалённая идентификация идёт через биометрию MyID. Локальный-первый подход — не ограничение, а самый быстрый к согласованию путь.

## 5. Почему это покупают банки
Главный страх банка перед ИИ — непрозрачная модель, принимающая кредитные решения. Здесь всё наоборот: детерминированно, объяснимо, **каждое решение за человеком**, **всё в журнале аудита**, **локально**. Это ИИ, который можно показать регулятору. ЦБ поддерживает контекст: **Стратегия ИИ до 2030 (PP-358)** прямо называет оценку кредитоспособности целевым применением; действует **регуляторная песочница** (для контролируемого пилота).

## 6. Предложение (следующий шаг)
Пилот (Шаги 3–4 метода): **on-site discovery** по процессам кредитования + AML → **выкатить одного агента за детерминированным шлюзом** → **предрегистрированная оценка на реальных данных** → аудит-трейл и шлюзы безопасности. Далее — продакшн (мониторинг, откат, SLA), затем регуляторный «последний метр» и суверенное развёртывание. Возможен запуск через **регуляторную песочницу**.

## 7. Как показывать демо (сценарий записи/показа)
1. Открыть «Кредит» → сюжет **МСБ** уже загружен (300 млн сум, 60 мес.).
2. «**Обработать заявку агентом**» → появляются контроль KYC/AML (метки чувствительных данных + data residency), комплектность 5/5, индикатор **DSTI 34% «в пределах политики»** с причинами и ссылками на документы, растёт журнал аудита.
3. Подчеркнуть шлюз: «**Решение — человек**». Нажать «**Одобрить**» → проект оферты + записи человека в аудите.
4. Переключить на **Физлицо** → «Обработать» → **DSTI 87% «вне политики»** → «**Отклонить**» → проект уведомления об отказе с основанием (порог 50%).
5. Финал — «**Журнал аудита**»: полная трассировка заявка → контроль → индикатор → решение человека → результат. «Локально, ничего не отправлено».

Скриншоты-раскадровка: `docs/bank-demo-screens/`. Запуск локально: `npm run serve` → http://127.0.0.1:5173 (вкладка «Кредит»).

## 8. Оговорки (честность = доверие)
- Демо — **синтетическое и локальное**; это доказательство управляющего цикла, а не готовая кредитная модель и не боевая интеграция.
- Регуляторные детали (дата ИИ-закона, точный набор норм DSTI, доступ к бюро, KYC/PEP-стек) — **перепроверить с банком и юристами** до обязательств.
- Не заявлять: отправку решений клиентам, автоматические кредитные решения, «полное соответствие» или «сертификацию».
