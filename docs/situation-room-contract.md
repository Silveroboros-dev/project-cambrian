# Situation Room Contract

Version: 0.1  
Date: 2026-06-30  
Scope: local synthetic Situation Room prototype  

## Purpose

The Situation Room shows how humans and active Cambrian agents coordinate around trusted Treuhand workflows.

It is not Slack, Gmail, browser monitoring, IAM, production telemetry, or autonomous execution. It is a local operating-room prototype with typed event cards, deterministic agent tags, governed work orders, approvals, evidence links, and audit records.

## Surfaces

1. **Rooms**
   Show local operating rooms: General Ops, Treuhand Case, Security, Onboarding, and Weekly Control.

2. **Timeline**
   Shows human messages, tagged work orders, and typed situation cards scoped to the selected `roomId`.

3. **Room Chat**
   Shows an append-only, room-scoped chat log. Each thematic room has its own retained message stream in local browser storage.

4. **Artifact Packs**
   Packs dense output into switchable views: agent cards, events, work orders, approvals, and local logs. Summary rows should be compact, with metadata available on expand.

5. **Demo Conductor**
   Runs local synthetic scenarios as numbered stakeholder-demo acts for inbound email intake, confidential upload attempt, employee onboarding, agent handoff gap, and weekly control audit. Each act must show what was created now and why the governed agent loop is stronger than a generic chatbot.

6. **Approvals**
   Shows approval requests generated by agents. Approval and rejection are human actions only.

7. **Guided Demo**
   Resets/prepares clean local demo state, runs the five synthetic scenarios in sequence, creates a recap card with agent/card/work-order/approval counts, and leaves a visible completion marker in the active Situation Room surface.

8. **Agent Participation**
   Shows the six active agents only: one Treuhand workflow agent and five control agents. Agent cards must be compact by default. Dense command and approval-boundary details must open in a centered overlay keyed by `agentId`, not by expanding the page flow. Clicking the same agent again collapses the overlay. Backlog agents must not be presented as built Situation Room participants.

9. **Situation Metrics**
   Shows Situation Room metrics bound to the current local store: messages, cards, work orders, approval decisions, blocked work, and log events. These metrics must update after review/approval actions.

10. **Demo Snapshots**
    Exports and imports versioned local demo-session snapshots so a second demo can resume a week later without adding a backend or database.

## Identity Keys

Every Situation Room artifact must carry explicit identity:

- `roomId` for room scope;
- `messageId` for human, agent, or system messages;
- `workOrderId` for tagged work orders;
- `cardId` for typed situation cards;
- `approvalId` for human approval gates;
- `caseId` when a workflow case is involved;
- `agentId` for the tagged or acting agent;
- `actorId` for the human/system actor;
- `traceId` for cross-card scenario traceability;
- `logId` for local event-log records;
- `snapshotId` for exported/imported demo-session snapshots;
- `evidenceIds` when a card references evidence.

## Agent Tags

The first local tag parser supports:

- `@treu` -> `A-TREU-001`;
- `@ingest` -> `A-INGEST-001`;
- `@sec` -> `A-SEC-001`;
- `@auth` -> `A-AUTH-001`;
- `@gap` -> `A-GAP-001`;
- `@cad` -> `A-CAD-001`.

Tagging an agent creates a governed work order. It must not directly execute sensitive external actions.

## Truth Labels

- Situation cards are local synthetic operating events unless explicitly sourced from a real adapter.
- Demo conductor artifacts carry `synthetic_local` / `synthetic/local only` truth labels.
- Work orders are proposed or running local tasks, not proof of autonomous production capability.
- Security, gap, and cadence cards are advisory unless a deterministic local rule marks them blocked.
- Approval requests are pending until a human approves or rejects them.
- Approval decisions are local approval records only. They do not send email, grant real access, or promote durable memory.
- Room context is case-local context, not approved reusable memory.
- Operating memory candidates remain proposed until human approval.

## Mutation Rules

Allowed local mutations:

- create messages, work orders, cards, and approval requests;
- append room-scoped chat messages and local log records;
- export and import versioned local demo snapshots;
- advance the synthetic demo by one week using prior local logs;
- run synthetic local scenarios;
- create local context packets, security findings, authorization decisions, gap findings, cadence nudges, audit events, and recommendations from scenarios;
- approve or reject local approval requests.

Forbidden mutations:

- poll real Gmail, Slack, Drive, or browser activity;
- send emails or external messages;
- grant real access or modify production IAM;
- monitor employees outside the synthetic demo surface;
- write durable operating memory without human approval;
- treat synthetic scenarios as real customer proof.

## Acceptance Criteria

### AC-SR-1: Situation Room Surface

The app must expose a `Situation Room` surface with room list, room-scoped timeline, scenario buttons, tag input, work orders, and approvals.

### AC-SR-2: Typed Artifact Model

Rooms, messages, work orders, cards, and approvals must preserve `roomId` and their own stable identity keys.

### AC-SR-3: Agent Tag Parser

The local parser must map `@treu`, `@ingest`, `@sec`, `@auth`, `@gap`, and `@cad` to the six active agents and create a work order with stages.

### AC-SR-4: Synthetic Scenario Gallery

The Situation Room must provide local scenario actions for inbound email intake, confidential upload attempt, employee onboarding, agent handoff gap, and weekly control audit.

### AC-SR-5: Scenario Artifacts

Scenario runs must create typed cards and compatible local artifacts where relevant: context packets, agent runs, security findings, authorization decisions, gap findings, cadence nudges, approval requests, and audit events.

### AC-SR-6: Human Approval Gates

Sensitive actions such as sending a draft, granting access, or approving memory/policy updates must create approval requests. Agents may propose; humans approve or reject.

### AC-SR-7: No External Side Effects

The prototype must not require or perform real Gmail, Slack, Drive, browser monitoring, IAM, LLM, or telemetry integration.

### AC-SR-8: Ambient Boundary

Ambient behavior is limited to notice, summarize, propose, and remind. It must not execute sensitive actions.

### AC-SR-9: Room Context Vs Operating Memory

Case room context may be visible inside the room. Reusable operating memory must remain a proposed candidate until explicit approval.

### AC-SR-10: Tests And Smoke Coverage

Automated tests and smoke checks must prove the contract exists, the UI entry point exists, agent tags create work orders, scenarios create cards and local artifacts, approval gates work, and no external service is required.

### AC-SR-11: Guided Demo

The Situation Room must provide a guided local demo action that runs inbound email intake, confidential upload attempt, employee onboarding, agent handoff gap, and weekly control audit in sequence from a clean local state.

### AC-SR-12: Active Agent Participation

The Situation Room must visibly summarize exactly the six active agents with latest action, cards, work orders, control outputs, and sensitive-action approval boundary.

### AC-SR-13: Demo Command Routing

The supported demo commands `@ingest process inbound email`, `@treu run intake review`, `@sec check upload`, `@auth onboard junior accounting assistant`, `@gap inspect handoff`, and `@cad run weekly audit` must create meaningful local artifacts, not only empty planned work orders.

### AC-SR-14: Local Approval Semantics

Approval resolution must update the approval card and work order status while stating that only a local approval record was created and no external email, access grant, or memory promotion occurred.

### AC-SR-15: Room-Scoped Endless Chat

Each room must have a retained chat stream scoped by `roomId`. Posting a message must append to that room only, and artifact-creating commands must append a visible system reply that names created artifact IDs.

### AC-SR-16: Packed Artifact Views

The Situation Room must pack dense state into switchable views for agent cards, events, work orders, approvals, and local logs. The active pack selection must be stored locally and must not change room identity.

### AC-SR-17: Review-Bound Situation Metrics

Situation metrics must update from current local state after approval/rejection and review actions. They must show at least total messages, cards, work orders, reviewed approvals, blocked work, and log records.

### AC-SR-18: Agent Next-Step Proposals

After a human resolves a sensitive approval, the acting agent must create a local next-step proposal card with explicit choices. The proposal is advisory and must not execute the choice.

### AC-SR-19: Local Log Store

Situation Room logs must be stored locally in `localStorage` as part of the app store under `situationEventLog`, alongside `roomMessages`, `situationCards`, `workOrders`, and `approvalRequests`.

### AC-SR-20: Timestamped Artifacts

Rendered messages, cards, work orders, approvals, and log events must show local timestamps so reviewers can reconstruct order of events.

### AC-SR-21: Versioned Demo Snapshot Export

The Situation Room must export a local JSON snapshot with `snapshotVersion`, `snapshotId`, `sessionId`, `createdAt`, `scenarioWeek`, and all local Situation/validation artifacts required to resume the demo.

### AC-SR-22: Versioned Demo Snapshot Import

The Situation Room must import only supported snapshot versions, restore the exported local artifacts, and record an import log without calling external services.

### AC-SR-23: Week-Two Continuity Scenario

The Situation Room must provide a local week-two scenario that reads prior local logs, summarizes unresolved approvals/blocked work/review actions, and creates new cards/logs proving continuity from the previous demo state.

### AC-SR-24: Compact Active Agent Cards

The active-agent panel must keep all six built agents visible without overwhelming the first viewport. Each agent row/card must show compact identity, role, latest action, and output counts by default. Dense demo-command and approval-boundary detail must open in a centered overlay bound to `agentId`; the compact grid must not push down content below it. Clicking the currently expanded agent must close the overlay.

### AC-SR-25: Demo Reset Visible Boundary

Resetting demo state must visibly state that it only prepared a clean local demo state. It must not claim to have created new agent work, and the user must run the numbered Demo conductor acts to create cards, work orders, approvals, and logs.

### AC-SR-26: Demo Conductor Per-Act Deltas

The Situation Room must present the stakeholder demo as numbered acts for mailbox intake, confidential upload, onboarding, handoff gap, and weekly audit. Clicking an act must create visible new local artifacts, update a `Created this act` delta panel with card/work-order/approval/log counts and artifact IDs, mark the act complete, and show a contrast between generic chatbot behavior and Cambrian governed-agent behavior. Resetting demo state must not pretend to create new agent work.

### AC-SR-27: Demo Conductor Readability

Demo conductor act titles must render as readable horizontal rows, not narrow vertical tiles that split words into fragments. Long artifact IDs in conductor and last-action surfaces must be compacted or contained so they do not overflow into adjacent panels; full traceability remains available in artifact details.

### AC-SR-28: Manual Tag Dedupe

Manual agent tags that match a Demo conductor act must route through the same idempotent act path. If the act was already completed, the tag may select and summarize the existing act but must not create duplicate cards, work orders, approvals, or logs. Only genuinely ad-hoc manual commands should create standalone planned work orders.
